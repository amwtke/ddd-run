# DDD Event Storming Skill

## 触发
用户使用 `/ddd-storm <业务描述>` 命令,或明确要求"事件风暴 / 业务梳理"。

## 目标
从一段自由的业务描述中,**不做实现**,只产出领域概念骨架。输出一份 Markdown 文档,
用作后续 `/ddd-model` 的输入。

## 产出格式

必须严格按以下结构输出中文 Markdown:

```markdown
# 事件风暴:<业务主题>

## 1. 业务描述复述
<用你自己的话把用户给的业务描述重述一遍,确保理解一致>

## 2. 参与者(Actors)
- **<Actor 名>**:<他/它在业务中扮演的角色>
- ...

## 3. 命令(Commands)
以"动词 + 名词"形式列出系统接收的命令。
- `<Command 名>`:由 <Actor> 发起,<做什么事>
- ...

## 4. 领域事件(Domain Events)
以"名词 + 过去分词"形式列出事实发生的事件。
- `<Event 名>`:<什么条件下发生,包含哪些关键信息>
- ...

## 5. 外部系统(External Systems)
- **<系统名>**:<交互点与职责>
- ...

## 6. 读模型 / 查询(Read Models)
- `<查询名>`:<谁会查,返回什么>
- ...

## 7. 聚合候选(Aggregate Candidates)
列出疑似聚合的概念,不要现在就下定论,留给 /ddd-model 细化。
- **<候选聚合名>**:理由,可能包含哪些内部实体/值对象
- ...

## 8. 开放问题(Open Questions)
主动提出你对业务规则的 2~5 个疑问,不要假设。
- Q1: ...
- Q2: ...

## 9. 下一步建议
> 建议运行 `/ddd-model` 基于本文档产出正式领域模型。
```

## 核心原则

1. **不写代码**。这一阶段只做概念抽取,连伪代码都不要。
2. **用业务语言,不用技术语言**。不要出现 `Entity`、`Repository`、`Service` 等词,
   只用"订单"、"会员"、"积分"这样的业务词汇。
3. **命令用现在时,事件用过去时**。`CreateOrder` vs `OrderCreated`。
4. **聚合候选只做"候选",不做"决定"**。边界的最终确定在 `/ddd-model`。
5. **必须提出开放问题**。这是展示你在思考业务而不是盲从需求的关键——漏掉的假设往往是 bug 的源头。

## 反例(绝不要这样写)

❌ "我会创建一个 `OrderService` 来处理 `createOrder` 方法"
❌ "使用 MySQL 存储订单数据"
❌ "需要引入 Redis 做积分缓存"

以上都是**技术实现**,与事件风暴无关。事件风暴只讨论**业务**。

## 与其他 skill 的衔接

- **上游**:用户自由的业务描述
- **下游**:产出的文档作为 `/ddd-model` 的输入
- **兄弟**:不要在此处调用 `/ddd-spec`(spec 生成发生在模型成型之后)

## 文件落位

产出写入:`docs/ddd/01-event-storming-<主题 slug>.md`

如果 `docs/ddd/` 不存在,先创建。
